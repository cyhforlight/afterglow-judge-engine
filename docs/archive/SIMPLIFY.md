# 项目重构指导原则（阶段性）

> 本文档是重构期的临时指导文档，记录当前阶段的特殊背景和重构重点。长期开发规范见 [AGENTS.md](./AGENTS.md)。

## 1. 开发者状态与项目阶段

### 开发者背景

- **技术背景**：计算机专业大学生，具备丰富的算法竞赛（ACM/OI）经历，对时空复杂度敏感，算法基础扎实
- **工程经验**：零工程开发经验，缺乏系统性架构思维，Go 语言尚未入门
- **当前阶段**：距离校招实习（Go 后端/中间件方向）面试仅剩一个月，本项目是核心冲刺项目

### 项目现状

- 基于 PRD 需求文档，前期通过 Vibe Coding（AI 辅助）快速生成了初版 Go 代码
- 代码具备基本功能，但充斥着"AI 生成味"（过度设计、冗余、缺乏深度思考）
- 当前处于重构期，目标是将代码重构为高标准、严要求的工程级代码

## 2. 项目**有可能**存在的问题

Vibe Coding 往往倾向于堆砌市面上常见的模板和设计模式，导致代码失去灵魂。典型问题包括：

### 2.1 冗余

- **重复代码**：多处相同或相似的逻辑未提取
- **重复计算**：同一个值被多次计算
- **重复抽象**：为同一个概念创建多个抽象层

### 2.2 臃肿

- **过度抽象**：为每个 model 都建 repository，实际只有简单 CRUD
- **空洞分层**：引入 manager/handler/processor 等多层包装，实际逻辑很简单
- **伪通用能力**：设计"可扩展框架"，实际只有一个实现
- **反向依赖**：config 依赖 service、model 依赖 transport 等

### 2.3 实现与预期不符

- **看起来像标准模板**：代码结构符合某种模板，但实际脱离项目场景
- **功能与定位不符**：设计了项目不需要的功能（如过度的配置化、通用化）
- **边界不清晰**：模块职责混乱，依赖关系复杂

## 3. 重构的开发规范

### 3.1 低耦合

- **单向依赖**：依赖方向保持单向（transport → service → model/sandbox/storage）
- **消除反向依赖**：config 不能依赖 service、model 不能依赖 transport
- **消除循环依赖**：任何两个模块之间不能形成循环依赖

### 3.2 准函数式

- **纯函数优先**：函数的输出只依赖输入，不依赖外部状态
- **不可变数据**：优先使用不可变数据结构，避免意外修改
- **避免副作用**：函数应该尽量避免副作用（如修改全局状态、I/O 操作）
- **高阶函数**：适度使用高阶函数（如 map、filter、reduce），但不过度

### 3.3 模块化

- **高内聚**：模块内部的功能应该紧密相关
- **清晰边界**：模块之间的边界应该清晰，职责明确
- **接口抽象适度**：便于测试，但不过度设计

### 3.4 现代性、优雅性、易学性

- **现代性**：优先使用 Go 1.22+ 的现代特性，避免旧模式（如 `for range N`、`slices.Contains`、`log/slog`）
- **优雅性**：架构的优雅、美观、整洁，码风的明了优雅，与功能实现同等重要
- **易学性**：代码应易读、易学习，让他人（包括未来的自己）能快速理解设计意图和实现细节

## 4. 第一性原理的两个层面

### 4.1 代码实现层面：是否需要这么复杂？

在审查每一段代码时，问自己：

- **这段代码的本质目的是什么？**
- **能否用更简单的方式实现？**
- **这个抽象层是否必要？**
- **这个函数/类/模块是否有其存在的绝对理由？**

### 4.2 功能主旨层面：是否需要这个功能？

在设计每一个功能时，问自己：

- **这个功能是否符合项目定位？**
- **这个功能是否真的需要？**（YAGNI 原则）
- **这个功能是否过度通用化？**（为假想的未来需求设计）
- **这个功能是否增加了不必要的复杂度？**

**原则**：
- 本项目定位为内部评测微服务，不是公网开放平台，不需要为"假想中的开放平台场景"额外堆叠过度复杂的安全设计
- 算法竞赛是一个需求较为稳定的领域，不需要为"可能的未来需求"设计可扩展框架
- 功能应该回归业务需求的本质
- 如果一个功能没有 “服务于项目核心功能的实现” 的任务，那么至少也应该 “能够服务于学习者对于后端 / 中间件 / Infra 等软件开发领域的学习”

## 5. 指导侧重点

在重构过程中，对于每一处需要修改的代码，重点在于：

### 5.1 剖析"坏味道的本质"

- **不仅指出问题**：说明"这段代码有问题"
- **更要说明原因**：说明"为什么有问题"、"问题的本质是什么"

### 5.2 给出"真实工程视角的最佳实践"

- **不是"看起来专业"的模板**：而是真正符合项目场景的实践
- **不是"算法竞赛的优化"**：而是工程上的可读性、可维护性
- **不是"炫技式的写法"**：而是直白、易懂的实现

### 5.3 启发开发者自行思考

- **提出问题**：引导开发者思考"为什么这样做"
- **对比方案**：展示不同方案的优劣

## 6. 与 AGENTS.md 的关系

- **AGENTS.md**：长期稳定的开发规范，是"宪法"
- **SIMPLIFY.md**：阶段性的重构指导，是"短期行政令"
- **重构完成后**：SIMPLIFY.md 可以归档，AGENTS.md 继续作为开发规范
