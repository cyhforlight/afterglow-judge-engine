# 项目开发指南

本指南记录 afterglow-judge-engine 的项目定位、系统模型和开发约定。若存在更高层级的通用规范，本指南只补充项目特有约束；单独阅读时，下述设计偏好仍作为默认判断方向，而不是必须机械套用的模板。

## 1. 项目定位与真实约束

本项目是一个学习型代码评测沙箱系统，定位为大型 OJ、命题系统或训练平台中的内部评测微服务，而非公网开放平台。

项目同时追求：

- 正确表达判题、资源限制和隔离语义
- 建立容易理解和演进的 Go 工程结构
- 让代码主要依靠结构、命名、类型和控制流说明自己，便于学习和维护
- 使用社区认可的现代稳定能力，不为不存在的旧技术栈增加兼容层

“学习项目”不意味着使用教学式脚手架或回避现代能力；“架构优雅”也不意味着追求目录对称和形式完整。这里的优雅首先指系统模型清楚、职责自然、普通变化集中且实现容易解释。

项目不承担旧 Go 版本或旧运行环境的兼容义务，但 HTTP API、外部资源格式和上游调用方仍可能构成真实契约。破坏这些契约前，应确认影响范围和迁移方式。

部分 sandbox、集成测试和 E2E 测试需要 root、containerd 及编译器工具链。权限是按需使用的运行前提，不是所有命令的默认执行方式。

## 2. 代码设计与实现偏好

### 2.1 结构与抽象

- 让每个领域概念拥有自然位置，按内聚关系和共同变化组织代码，不因文件长度或目录对称机械拆分。
- 边界和层次应吸收真实复杂度，例如隔离 containerd、翻译外部语义或集中稳定规则；没有隐藏内容的转发层不值得存在。
- 保持完整判题流程清楚连贯，让缓存、并发限制、workspace 等辅助机制待在相应边界内。
- 只提取真实、稳定且名字准确的共性。变化方向未知时保持实现具体，避免为未来需求预设开关、扩展点和通用框架。
- 当普通修改持续扩散到多个无关位置时，重新检查职责和知识归属，而不是继续增加局部补丁。

### 2.2 表示与明确性

- 优先寻找能够消除状态和分支的表示。善用不变量、定义表、阶段类型和可推导关系，让无效组合尽量难以产生。
- 相同事实和规则只定义一次；明确呈现会改变行为的语义选择、重要依赖、控制流和失败路径。
- 内部代码信任已经建立的契约，不为理论上的误用重复堆叠校验和兜底。
- 控制流保持视线清晰，尽早处理错误和特殊情况，避免不必要的嵌套。
- 注释说明原因、约束和反直觉事实，不复述代码表面行为。

### 2.3 演进与技术选择

- 没有外部兼容义务、迁移范围可控且能够回滚时，可以主动修正错位结构，必要时允许破坏性重构或重写。
- 优先使用当前 `go.mod` 支持的现代稳定 Go 能力。选择标准是减少整体复杂度并容易说明，而不是单纯追新或刻意使用基础语法。
- 小问题优先标准库或项目已有能力；通用且难以写对的机制优先成熟依赖。同一职责尽量不引入第二套工具。
- 依赖和抽象的重量应与其吸收的复杂度相称。性能满足现实约束后，不为未经验证的余量增加状态和边界情况。
- 判题时间、内存、输出限制以及隔离强度属于领域正确性，不应被当作普通性能优化随意弱化。

## 3. 系统模型与依赖方向

当前实现采用一条克制的分层链路：

- `transport/httptransport`：负责 HTTP 路由、鉴权、请求体大小限制、JSON 解码、调用 service 和响应编码
- `service`：负责完整判题流程编排，包括加载测试数据、解析 checker、编译、执行、校验、汇总逐点结果和判题流程状态
- `execution`：负责通用容器执行任务，包括准备 workspace、写入文件、表达资源限制、调用 sandbox、收集产物和限制容器并发
- `sandbox`：负责通过 containerd 在受限环境中执行编译和运行动作
- `resource`：负责内部资源和外部文件的只读访问
- `model`：负责判题请求、结果和枚举类型，并承载当前 HTTP API 的 JSON 字段约定

依赖方向保持单向：

```text
transport -> service -> model
                    -> resource
                    -> execution -> sandbox
```

不能出现反向依赖或循环依赖。该结构描述当前有效的系统模型，不是不可调整的目录模板；如果职责或变化方向发生实质改变，应连同本节一起更新。

`model` 当前直接承载 JSON 约定，是单一 HTTP 接口下的有意简化。只有当传输模型与判题模型开始独立变化，或出现第二种传输方式时，再拆分 transport DTO。

### 接口使用

- 边界不等于 Go interface，不为每个 struct 机械定义接口。
- 接口可以表达真实的多实现、外部依赖替换、能力阶段或有价值的测试接缝；即使当前只有一个实现，也应说明它隐藏了什么变化或约束。
- 不单纯为了生成 mock 改造生产设计，也不保留只做方法转发的形式化接口。
- 接口通常由使用方按所需能力定义，并保持小而具体。

## 4. 信任边界与 Go 约定

### 信任边界

HTTP 请求、外部文件和路径、用户提交的程序、编译器输出以及 containerd/OCI 返回值都属于不可信边界。应在最接近边界的位置完成大小限制、格式校验、路径约束和外部语义翻译。

一旦数据已经转换为满足不变量的内部表示，后续代码应信任该契约。安全校验集中在拥有相关知识的位置，避免各层重复实现略有差异的检查。

### 错误与日志

- 错误是正常控制流的一部分，不得静默吞咽。
- 静态错误使用 `errors.New`；需要动态信息或包装下层错误时使用 `fmt.Errorf`，包装时保留 `%w`。
- 错误消息说明当前操作和必要上下文，不在每一层重复添加同义包装。
- 需要记录日志时统一使用 `log/slog`。由能够解释或处理失败语义的边界记录一次，避免同一错误沿调用链重复记录。
- HTTP 接口统一返回约定的 JSON 响应；具体编码方式由 transport 层负责。

### 命名、控制流与并发

- 小作用域且含义明确时使用 `i`、`n`、`ctx`、`err`、`ok`、`buf` 等短名称；存在歧义时使用能够表达角色和单位的具体名称。
- 避免遮蔽预声明标识符，尤其是在影响理解或阻止调用内置能力时；机械规则以 lint 配置为准。
- 优先直线化控制流和明确的错误路径，不用晦涩技巧换取局部代码长度。
- 并发只用于吸收真实并行需求或资源调度约束。优先通过消息传递、所有权和局部状态减少共享可变状态，避免滥用全局状态和锁。

## 5. 技术栈与工具链

- Go 版本以 `go.mod` 为唯一事实来源，当前为 Go 1.25.0。
- 目标运行环境为 Ubuntu 22.04、Debian 12 或更新版本。
- 格式化和 import 管理使用 goimports。
- 代码质量检查使用 golangci-lint，具体规则以 `.golangci.yml` 为准。
- 测试使用标准库 `testing` 和现有的 testify；`assert` 用于可继续执行的比较，`require` 用于后续步骤依赖的前置条件。

## 6. 测试策略

### 测试风格

- 多个案例共享相同执行路径和断言结构时使用表格驱动测试；具有独立流程、前置条件或失败语义的场景使用具名测试。
- 优先覆盖领域不变量、信任边界、失败路径和真实临界条件，不机械枚举与语义无关的零值、负数或实现细节。
- 核心逻辑以 70% 以上覆盖率为观察目标，但不通过低价值测试凑数，也不因重构把关键路径测空。
- 测试应验证对外可观察行为；只有在内部表示本身承载重要不变量时，才直接测试内部细节。

### 集成测试与 E2E

- 默认测试命令运行完整测试集，不依赖 `testing.Short()` 跳过集成测试或 E2E 测试。
- 对 root、containerd、`gcc`、`g++`、`javac`、`jar` 等真实运行前提进行显式检查。环境不具备时可以 `Skip`，但不能把环境问题伪装成业务失败。
- `Skip` 表达“当前环境无法执行”，不能仅用于缩短默认测试时间。
- 环境相关测试应把前置条件留在测试代码中，使约束清楚可见。

## 7. 质量门禁

所有代码修改在交付前应满足：

1. 对本次修改的 Go 文件运行 goimports。
2. `golangci-lint run ./...` 通过，且不引入新的 lint 问题。
3. 相关测试通过；环境允许时运行 `go test -cover ./...`。
4. 修改 `.golangci.yml` 时额外运行 `golangci-lint config verify`。
5. 因环境前置条件无法完成的验证必须明确说明。

若上级环境要求通过命令代理执行工具，例如 `rtk`，应为下列底层命令添加相应前缀。

```bash
golangci-lint run ./...
go test -cover ./...
```

## 8. AI 辅助开发

AI 用于加速阅读、设计、实现和验证，不替代工程判断。生成的代码必须回到真实需求、系统模型和质量门禁上重新审视。

重点警惕以下问题：

- 为单一实现镜像接口或提前设计扩展框架
- 添加没有吸收复杂度的 manager、processor 等转发层
- 在多个层次重复定义相同规则、DTO 或配置
- 为理论误用增加无依据的校验和兜底
- 引入反向依赖，或让普通修改持续跨越多个无关模块

发现这些迹象时，优先恢复对真实差异和主要流程的直接表达，而不是继续补充脚手架。
